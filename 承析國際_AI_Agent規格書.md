# 承析國際貿易有限公司：AI Agent 規格書（可驗證實作版）

版本：v0.1｜日期：2026-07-15｜狀態：供立項與 PoC 驗收

## 1. 截圖分析與結論

截圖所描述的公司情境為化學原料、粉末冶金、汽機車／自行車零件與五金機械的國際貿易。痛點集中在：海外客戶開發、多國語言商務往來、TDS/SDS/RFQ/報價單等文件的理解與比對，以及發票、裝箱單、提單與報關資料的抄打。製程參數優化亦被提及，但它需要可用、可追溯的歷史製程與良率資料，**不應列入第一期**。

建議不是採購多個互不相通的 AI 網站，而是建立一個「貿易業務工作台」：以既有 CRM、郵件與 ERP 為正式紀錄來源；AI 只讀取授權資料、產出有來源可追查的草稿與結構化欄位；對外寄信、報價、下單、報關送件一律由人員核准。此做法符合截圖的「前段 AI 自動化、後段業務接手」思路，也避免錯誤內容直接影響客戶與交易。

## 2. 成功定義與範圍

### 2.1 第一階段商業目標

1. 將合格潛客研究、首封英文信草稿與 CRM 建檔所需時間降低 50% 以上。
2. 將收到 RFQ 到形成「可覆核的需求摘要」的中位處理時間降低 60% 以上。
3. 文件欄位（料號、品名、數量、單位、幣別、價格、Incoterms、交期、有效期）在人工複核後達 99% 正確；未覆核前不寫入 ERP。
4. 所有 AI 結論都能顯示原始郵件、文件頁碼／段落或資料來源；無來源者只能標示「待確認」。

### 2.2 In scope（MVP）

| Agent | 解決問題 | 交付物 | 是否可自動執行 |
|---|---|---|---|
| A1 市場與名單研究 Agent | 找客戶、判斷是否符合 ICP | 公司卡、聯絡人候選、來源連結、適配分數 | 可存「待覆核名單」；不可自動寄信 |
| A2 開發信與跟進 Agent | 客製英文／日文開發信、避免漏跟進 | 信件草稿、跟進時程、CRM 任務 | 建草稿與任務；寄信需核准 |
| A3 商務溝通與翻譯 Agent | 回覆外語詢問、維持術語一致 | 雙語回信草稿、術語提示、風險旗標 | 僅草稿 |
| A4 RFQ／規格知識 Agent | 從 TDS、SDS、圖面、RFQ 擷取並問答 | 需求摘要、規格表、引用頁碼、缺漏清單 | 可建立案件草稿；不可承諾規格／價格 |
| A5 貿易文件審核 Agent | 降低發票／裝箱單／提單抄打與不一致 | JSON 欄位、跨文件差異、ERP 匯入草稿 | 不可過帳、送報關或付款 |
| A6 供應商報價比對 Agent | 多家報價難比較 | 正規化比較表、成本／交期／MOQ／付款條件差異 | 僅建議採購方案 |

### 2.3 Out of scope（第一階段不做）

- 自動爬取或大量寄送冷郵件、繞過網站或 LinkedIn 的使用規範。
- AI 自主決定對外報價、變更化學材料規格、簽約、付款、報關申報或下採購單。
- 製程最佳化模型。第二期前提是至少有一個明確製程的時間序列、品質檢驗結果、批次 ID 和足夠歷史樣本，且能由製造／品保確認資料品質。

## 3. 使用者、權限與責任

| 角色 | 可做的事 | 不可做的事 |
|---|---|---|
| 業務 | 查詢、建立草稿、核准自己名下的開發信 | 修改核准後的文件紀錄、對外報價免覆核 |
| 業務主管 | 設定 ICP、核准群發批次、查看 KPI | 存取非授權業務的敏感附件 |
| 採購／貿易人員 | 上傳文件、複核欄位與比價 | 由 AI 直接過帳或報關 |
| 品保／技術 | 維護材料術語、批准規格答案 | 讓模型自行改寫受控規格 |
| 系統管理員 | 串接帳號、權限、保留期限、稽核 | 閱讀模型以外的未授權商務內容 |

所有寫入動作使用 `draft → human_approved → executed` 三態。Agent 沒有「直接寄出／過帳」權限；每次核准必須記錄核准人、時間、輸入版本、輸出版本及理由。

## 5. 功能規格

### A1 市場與名單研究 Agent

**輸入**：目標地區、產業／應用（例如汽車零件、粉末冶金）、材料／產品能力、排除條件、客戶既有名單。來源限定為公司已訂閱且授權使用的資料庫、官方公司網站與使用者提供的清單。

**處理規則**：

1. 去重：以網域、公司法定名稱、地址近似度比對；與 CRM 現有帳戶衝突時標示「既有客戶／待合併」。
2. ICP 評分（0–100）：產業適配 30、地區／貿易可達性 15、產品／材料適配 25、規模／採購訊號 15、資料可信度 15。低於 70 不產生外聯建議。
3. 每個判斷附來源 URL、抓取日期與證據文字。不可把推論包裝成已驗證事實，例如「可能使用銅粉」必須標為推測。

**輸出契約**：`company_name, domain, country, application, evidence[], icp_score, score_reasons[], contacts[], crm_match_status, reviewer_status`。聯絡人 email 必須標示 `source_verified / provider_verified / unknown`；`unknown` 不得進入寄送名單。

### A2 開發信與跟進 Agent

**輸入**：覆核後公司卡、產品能力庫、語言、業務署名、已寄信／回覆紀錄。

**輸出**：主旨三案、120–180 字信件草稿、個人化依據、CTA 一個、建議跟進日與停止條件。內容只能引用能力庫中已核准的產品、認證和案例；禁止虛構庫存、認證、價格、MOQ 或交期。

**寄送防護**：

- 第一期採「一封一核」或業務主管核准的名單批次；不採無上限自動群發。
- 預設最多 2 次跟進；收到拒絕、退信、退訂或負面回覆，立即封鎖後續訊息並同步 CRM。
- 寄送前檢查公司網域、收件人、重複寄送間隔、抑制名單與必填退訂／聯絡資訊（依法務確認適用市場後啟用）。

### A3 商務溝通與翻譯 Agent

來信進入後先分類 `RFQ / 規格詢問 / 樣品 / 議價 / 物流單據 / 負向回覆 / 其他`，給出信心分數與建議負責角色。翻譯採「受控術語表優先」：料號、牌號、法規名、Incoterms、尺寸與單位不改寫；所有數字、單位、日期、幣別用規則引擎比對原文。低信心、含價格或技術承諾的草稿必須由相應角色核准。

### A4 RFQ／規格知識 Agent

文件上傳後依序：惡意檔掃描 → OCR／版面擷取 → 文件分類 → 欄位擷取 → 來源定位 → 規則驗證 → 人工複核。問答時只檢索該 RFQ 案件與獲授權的產品知識庫；答案需帶「文件名稱、頁碼、段落／表格列」，找不到證據時答「文件未載明」。

**需求摘要的必填欄位**：客戶、料號、材料／牌號、化學成分、粒徑／尺寸、密度、外觀、數量、單位、交貨地、目標交期、品質／測試要求、包裝、認證／法規、報價幣別與 Incoterms。缺值、互相矛盾或可信度小於 0.90 時列為「待人工確認」，不可自行補值。

### A5 貿易文件審核 Agent

支援 Invoice、Packing List、B/L、提單、報關資料與供應商報價。以單據號碼、料號、數量、單位、毛／淨重、箱數、幣別、單價、總額、HS Code、起訖港、Incoterms、日期建立結構化資料；跨文件規則例如 `Invoice quantity = Packing List quantity`、總金額＝各列數量×單價（以容許的四捨五入誤差）、重量與箱數不可為負。輸出「通過／需覆核／阻擋」與欄位層級來源。永遠只能產生 ERP 匯入草稿。

### A6 供應商報價比對 Agent

將報價標準化為相同幣別、單位與交期基準；匯率使用財務指定的日期與資料源，不能由模型猜測。比較表至少包含：符合規格、單價與總成本、幣別、MOQ、包裝、交期、付款條件、有效期、產地、品質文件和偏離項。推薦結果必須把「最低價」和「符合規格／交期」分開呈現，並列出推薦前提與未驗證資訊。

## 6. 技術架構與可實現性

### 6.1 元件

- **工作台**：瀏覽器介面，供搜尋、上傳、檢視引用、覆核與審計。
- **Agent 服務**：以具結構化輸出與工具呼叫能力的 LLM 為編排層；模型不得直接連資料庫或外部系統，必須經過受控工具 API。
- **規則與工作流服務**：負責狀態機、數字校驗、重試、核准與通知；這些確定性工作不交由模型判斷。
- **資料層**：關聯式資料庫儲存交易欄位與稽核事件；加密物件儲存原始文件；向量索引僅存已核准／有 ACL 的可檢索內容。
- **整合層**：CRM、公司郵件、ERP、文件庫、OCR／翻譯 API 以 OAuth／服務帳號連接；祕密放在祕密管理服務，不出現在提示詞、日誌或程式碼。

### 6.2 已核對的技術可行性

1. HubSpot 的 workflow 可呼叫 JavaScript 或 Python 自訂程式碼，也可將資料送到外部服務；但執行有 20 秒與 128 MB 限制。因此 CRM 只負責觸發、讀寫欄位與任務；文件解析改由非同步工作佇列處理。[HubSpot 官方文件](https://developers.hubspot.com/docs/api-reference/latest/automation/workflow-actions/custom-code-actions)
2. 若公司使用 Microsoft 365，Microsoft Graph 可建立草稿、寄信、讀取授權郵件與訂閱新郵件通知；`sendMail` 需要 `Mail.Send`。本規格刻意以草稿和人工核准包住寄信權限。[郵件自動化](https://learn.microsoft.com/en-us/graph/outlook-create-send-messages)、[sendMail 權限](https://learn.microsoft.com/en-us/graph/api/user-sendmail?view=graph-rest-1.0)、[郵件變更通知](https://learn.microsoft.com/en-us/graph/api/subscription-post-subscriptions?view=graph-rest-1.0)
3. DeepL 文件 API 可處理 DOCX、XLSX、PDF、PPTX、HTML、TXT 等格式，並支援 glossary；適合作為翻譯工作流的可替換供應商。其 PDF 轉換或 OCR 失敗仍須走人工覆核，不能假設排版與數字永遠正確。[DeepL 文件翻譯 API](https://developers.deepl.com/api-reference/document)
4. 因此上述 MVP 所需的 CRM、郵件、文件翻譯與外部工作流串接皆有成熟 API 路徑。**尚未驗證的是公司內部 ERP 是否有 API／匯入格式、現有郵件與 CRM、以及文件量與語言組合；這三項是啟動前必做的 discovery。**

## 7. 資料、資安與治理

- 採最小權限：郵件先使用指定共用信箱，不給全公司信箱讀取權；文件庫依客戶／案件 ACL 檢索。
- PII、報價與化學／貿易文件加密傳輸與靜態加密；設定地區、保留年限與刪除流程。送模型前遮罩非必要個資。
- 每次回答與欄位擷取記錄：文件雜湊、版本、檢索片段 ID、模型版本、提示模板版本、工具呼叫、使用者與核准事件。
- 防提示注入：附件／郵件內容一律視為不可信資料，禁止其中指令改變系統規則、呼叫工具或外傳資料；只允許白名單工具與固定 JSON schema。
- 建置「禁止自動承諾」規則：價格、交期、化學成分、法規適用、付款與報關欄位均須人簽核。高風險檔案或客戶資料不得用個人免費帳號上傳。

## 8. PoC 驗證與驗收標準

PoC 以 20 家目標公司、30 封歷史往返信件、30 份去識別化文件（至少涵蓋 RFQ、TDS/SDS、Invoice、Packing List、供應商報價）進行；資料先由資深業務／貿易人員標註黃金答案。不得只以模型自評作為驗收。

| 測試 | 通過標準 |
|---|---|
| 名單去重 | 與 CRM 已知重複帳戶的召回率 ≥95%，誤合併率 ≤2% |
| ICP 評分 | 由業務判為合格的前 20 名中，≥80% 被評為 70 分以上；每筆均有來源 |
| 信件草稿 | 20 封中，100% 不虛構公司能力／認證／價格；≥80% 僅需輕微編修即可使用 |
| 回覆分流 | RFQ、拒絕、一般詢問的分類正確率 ≥90%；拒絕類 100% 加入抑制名單 |
| 翻譯保真 | 料號、數值、單位、日期與幣別保留正確率 100%；商務用語由人工盲測 ≥4/5 |
| RFQ 擷取 | 必填欄位 precision ≥98%、recall ≥90%；每個已填值都有正確頁碼證據 |
| 單據勾稽 | 人為植入的 20 個數量／金額／條件差異，≥19 個被攔截；不得產生假陽性「已通過」 |
| 人工閘門 | 嘗試直接寄信、過帳、送報關均應被拒絕並留下稽核事件 |
| 權限隔離 | 使用者無權限時，搜尋、引用、模型回覆均不得洩露另一客戶案件內容 |
| 韌性 | OCR／ERP／模型服務失敗時工作進入可重試佇列，不重複建立 CRM 紀錄或重複寄信 |

每個 KPI 都需有導入前兩週基線。正式上線門檻：所有安全與人工閘門測試通過、文件與寄信類關鍵錯誤為 0、業務／貿易主管書面簽核測試報告。

## 10. 啟動前必答清單

1. 現有 CRM、ERP、郵件系統和文件庫是什麼？是否有 API、CSV 匯入或自訂欄位？
2. 哪些產品能力、認證、材料術語、標準品名與禁售／禁運規則是唯一正式版本？由誰維護？
3. 哪些國家市場、寄信頻率與退訂規範由法務／主管確認？
4. 哪些客戶、供應商與技術文件不得送入外部模型或翻譯服務？資料要落在哪個區域、保留多久？
5. 誰有權核准：開發信、技術規格答覆、報價、ERP 匯入與報關資料？

只要上述五項有答案，即可把本規格轉成 backlog、資料欄位與可執行的 PoC 測試案例；沒有答案時，Agent 仍可做「草稿與摘要」，但不應接觸正式交易流程。
